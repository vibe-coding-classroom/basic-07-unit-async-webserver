針對 **`basic-07-unit-async-webserver` (ESP32 Web Server 函式庫入門)** 單元，這是學員從「硬體開發」跨入「嵌入式網頁工程」的技術分嶺嶺。它要求學員具備 **事件驅動 (Event-Driven)** 與 **解耦設計 (Decoupling)** 的思維，學習如何利用 C++ Lambda 語法與範本引擎 (Template Engine)，在不阻塞馬達控制循環的前提下，打造一個具備即時互動能力的 Web 儀表板。

以下是在 **GitHub Classroom** 部署其作業 (Assignments) 的具體建議：

### 1. 範本倉庫 (Template Repo) 配置建議
非同步伺服器作業的核心在於「路由邏輯」與「參數安全」，範本應包含基礎框架與靜態資源，建議包含：
*   **📂 `src/main.cpp`**：**伺服器路由實作區**。學員在此定義 `server.on()` 處理邏輯，實作 Lambda 匿名函式、參數解析與處理。
*   **📂 `include/html_assets.h`**：**靜態網頁資源**。使用 `R"rawliteral(...)"` 宣告包含預設佔位符（如 `%UPTIME%`）的 HTML 字串，方便學員進行範本實作。
*   **📂 `platformio.ini`**：**依賴配置中心**。需預先配置好 `ESPAsyncWebServer` 與 `AsyncTCP` 函式庫庫，避免學員因環境配置問題受挫。
*   **📂 `README.md`**：**伺服器開發報告**。提供表格讓學員記錄：「API 路徑」 vs 「預期行為」 vs 「HTTP 狀態碼」，並要求截圖展示透過網址列控制硬體的成果。

### 2. 作業任務部署細節

#### 任務 1：「非同步首頁」第一個 Web 介面 (Async Landing Page Lab)
*   **目標**：掌握非同步路由架設，理解為何在通訊時「不佔用 loop()」是開發機器人控制系統的生命線。
*   **Classroom 部署建議**：
    *   **路由對準**：建立根路徑 `/` 路由，回傳 `include/html_assets.h` 中的頁面。
    *   **系統狀態檢視**：在頁面上整合 `ESP.getFreeHeap()`，讓學員觀測並證明「伺服器連線中，背景系統資源依然分配正常」。
    *   **驗證要點**：瀏覽器能否成功連線？刷新網頁時，硬體端的 Serial 輸出是否流暢（無延遲）？

#### 任務 2：「互動控制器」URL 參數解析 (Interactive Control Lab)
*   **目標**：學會從 URL 位址列提取動態參數，並實作 **防禦性參數檢查**，防止非法數值造成系統崩潰。
*   **Classroom 部署建議**：
    *   **命令接口實作**：建立 `/set_led?value=128` 路由。學員需解析 `value` 並將其套用至 LED PWM。
    *   **安全邊界檢查**：**核心關鍵點**。學員必須使用 `request->hasParam("value")` 檢查參數是否存在，並對數值進行 0-255 的限制。若無效，必須回傳 `400 Bad Request`。
    *   **驗證要點**：**關鍵技術審查**。直接存取無參數的網址是否會造成 ESP32 重啟？（若學員未做防錯處理，程式會因存取空指標而當機）。

#### 任務 3：「動態佔位符」範本引擎 (Template Engine Lab)
*   **目標**：掌握伺服器端渲染 (SSR) 的雛形，學習如何將「顯示外殼」與「動態數據」優雅分離，提升代碼可維護性。
*   **Classroom 部署建議**：
    *   **Processor 回呼設計**：實作一個 `processor` 函式，專門處理網頁字串中的佔位符。
    *   **數據動態注入**：當網頁偵測到 `%UPTIME%` 或 `%STATUS%` 時，由 Processor 自動替換為硬體真實運作時間。
    *   **最終 Commit**：封裝完整的「邏輯與內容分離」專業版本並 Push。
    *   **驗證要點**：網頁上的數據是否能隨刷新而動態更新？HTML 代碼中是否依然保持整潔（無夾雜 C++ 變數拼接）？

### 3. 指令架構與安全合規點評標準 (API Architecture & Safety Standards)
此單元的價值在於 **「對硬體狀態的 Web 化抽象能力與對通訊流程的非阻塞掌控」**：
*   [ ] **非阻塞設計執行**：路由處理是否完全使用回呼機制，確保 `loop()` 函數保持極限輕量？
*   [ ] **參數解析安全性**：是否實作了「存在性檢查 (hasParam)」？這對嵌入式設備的穩定性至關重要。
*   [ ] **範本解耦優雅度**：是否理解並運用了 Processor 機制，避免在 `main.cpp` 中出現大量的字串手動拼接？

